# Demux

## Getting started

### Introduction

The Demux module is designed to demultiplex and process raw reads generated by AXELIOS, the Roche SBX sequencing
platform.
It supports demultiplexing, adapter trimming, duplex pairwise intramolecular consensus with YC tag generation, and
filtering.

Demux consists of two command line interface (CLI) tools:

1. `demux` - performs demultiplexing, trimming, and filtering of raw sequencing reads. It handles both simplex and
   duplex data and on duplex data types performs pairwise intramolecular consensus and YC tag generation.
2. `demux_strand_index` - builds a strand index (a modified Bloom filter) for optional genome strand detection of duplex
   reads in `demux`. May not apply to many applications, see [Genome Strand Detection](#genome-strand-detection) for
   more information.

#### Recommended System Requirements

- Processor - Modern processor with at least 16 cores. A processor supporting `AVX2` or `AVX-512` will result in much
  better performance.
- Memory - For `demux`: 8 GiB. For `demux_strand_index`: 64 GiB.

#### Minimum Memory Requirements

- `demux` memory scales based on thread usage and parameters used such as `--batch-size`. Using default parameters,
  allocate at least 1 GB per 10 threads. If using strand detection, the memory increases by file size of strand index
  used.
- `demux_strand_index` scales according to the size of reference used. Around 1 Gbp will take at least 10 GiB of memory.
  For example, the human genome at around 3 Gbp would require around 32 GiB of memory and produces a 2 GiB strand index
  using default parameters.

## Usage

The following sections describe the procedures for demultiplexing raw simplex and duplex sequencing reads, and building
a strand index for genome strand detection.

### Demultiplexing raw _simplex_ sequencing reads

This is implemented as a structured pipeline in `demux` with the following steps:

1. **Gather input files** in FASTQ or RDB format.
2. **Demultiplex input reads** into different samples using the provided sample sheet and adapter design.
3. **Trim adapters** from reads according to the specified adapter design.
4. **Filter reads** based on length, error rate, and other specified criteria.
5. **Write processed reads** to the output directory.
6. **Calculate metrics** and write summary output files.

Components such as filtering may be performed slightly out of order for algorithmic or performance reasons.

#### Example command

```bash
demux \
  --adapter-design-bundle /resources/adapter-design-bundle.zip \
  --adapter-design-name YSU-SL \
  --threads 16 \
  --sample-sheet /path/to/sample_sheet.csv \
  --input /path/to/input_directory \
  --out-dir /path/to/output_directory
```

This command demultiplexes raw reads from the input directory `/path/to/input_directory`, using the adapter design
bundle located at `/resources/adapter-design-bundle.zip` and the adapter design name `YSU-SL`.
The output files will be written to `/path/to/output_directory`, using 16 threads for processing.
Sample names will be derived from the sample sheet. Reads for each sample will be written to its own subdirectory
(named for the sample) under the output directory.

For more details on the available command line options, see [Overview and CLI Options](#overview-and-cli-options).

#### Input files (simplex `demux`)

1. Adapter design bundle
2. Input raw reads FASTQ or RDB files or directories containing these files
3. Optional sample sheet file. If not provided, the sample sheet from the adapter design bundle will be used.
    - See section [Sample Sheet Format](#sample-sheet-format) for more details.

#### Output files (simplex `demux`)

1. Demultiplexed FASTQ files for each sample
2. Metrics files in TSV format
    1. Run metrics
    2. Sample metrics
    3. Sample assignment metrics
    4. Read length distribution metrics ( multiple files see below)

##### Read length distribution metrics

Read length distribution metrics is split into 4 files, one for each read type:

1. `trimmed_full_read_len_dist.tsv`

### Demultiplexing raw _duplex_ sequencing reads

This is implemented as a structured pipeline in `demux` with the following steps:

1. **Gather input files** in FASTQ or RDB format.
2. **Demultiplex input reads** via the hairpin specified by the adapter design and sample sheet.
3. **Perform pairwise intramolecular consensus** for duplex reads also generating YC tags.
4. **Trim adapters** from reads according to the specified adapter design.
5. **Filter reads** based on length, error rate, and other specified criteria.
6. **Write processed reads** to the output directory.
7. **Calculate metrics** and write summary output files.

Components such as filtering may be performed slightly out of order for algorithmic or performance reasons.

Example command:

```bash
demux \
  --adapter-design-bundle /resources/adapter-design-bundle.zip \
  --adapter-design-name SBX-D \
  --threads 16 \
  --sample-sheet /path/to/sample_sheet.csv \
  --input /path/to/input_directory \
  --out-dir /path/to/output_directory
```

This command demultiplexes raw reads from the input directory `/path/to/input_directory`, using the adapter design
bundle located at `/resources/adapter-design-bundle.zip` and the adapter design name `SBX-D`.
The output files will be written to `/path/to/output_directory`. 16 threads are allocated for processing.
Sample names will be derived from the sample sheet. Reads for each sample will be written to its own subdirectory
(named for the sample) under the output directory.

For more details on the available command line options, see [Overview and CLI Options](#overview-and-cli-options).

#### Input files (duplex `demux`)

1. Adapter design bundle
2. Input raw reads FASTQ or RDB files or directories containing these files
3. Optional sample sheet file. If not provided, the sample sheet from the adapter design bundle will be used.
    - See section [Sample Sheet Format](#sample-sheet-format) for more details.

#### Output files (duplex `demux`)

1. Demultiplexed FASTQ files for each sample

### Build an index for reference genome strand detection

`demux_strand_index` builds an interleaved Bloom filter for alignment-free genome strand detection of reads in `demux`.

The Bloom filter serves as a memory-efficient storage of k-mers from the reference genome, allowing for fast membership
queries during strand detection.

We interleave two Bloom filters, one for the forward strand and one for the reverse strand, to optimize memory access
patterns during querying.

Example command:

```bash
demux_strand_index \
  --input /path/to/reference_genome.fa \
  --threads 16
```

This command creates a Bloom filter file `strand.bloom` in the current directory, using 16 threads, and the reference
genome file located at `/path/to/reference_genome.fa`.

`demux` can then accept this Bloom filter file to enable strand-detection during demultiplexing.

```bash
demux \
  --adapter-design-bundle /resources/adapter-design-bundle.zip \
  --adapter-design-name SBX-D \
  --threads 16 \
  --input /path/to/input_directory \
  --out-dir /path/to/output_directory \
  --strand-detect strand.bloom
```

For more details on the available command line options, see [Overview and CLI Options](#overview-and-cli-options).

#### Input file for `demux_strand_index`

The reference genome FASTA file. The associated FASTA index file (`*.fa.fai`) must also exist.

#### Output file for `demux_strand_index`

The Bloom filter file with the default file name `strand.bloom` in the current directory. The file name and or path can
be changed using the `--output` option.

#### Tips for `demux_strand_index`

1. The Bloom filter only needs to be built once and can be reused for the same reference genome fasta.
2. Increasing the k-mer size (via `--kmer-size`) may increase the precision of strand detection, but strand detection
   becomes less tolerant to sequencing errors.
3. Lowering the max false positive rate (via `--max-false-positive-rate`) may reduce Bloom filter false positives during
   strand detection but increase the memory usage (and file size) of the Bloom filter.
4. Bloom filter entropy optimal FPRs are 1/2^n, where n is an integer and the corresponding optimal number of hash
   functions. The default value of `0.015625` corresponds to `n=6` which is the default. To keep your life simple do not
   change `--max-false-positive-rate` unless you have memory issues when running with `demux`.
5. The filter will be built in sizes of powers of 2, so picking a `--max-false-positive-rate` that optimizes for this
   depends on the size of the reference genome. For a human genome, it will produce a 2 GiB filter with the default
   value of `0.015625` and just so happens nearly at the optimal information density (entropy).

#### Performance and cost notes for `demux_strand_index`

A custom hash table is used to find unique k-mers. Though we mentioned the memory usage to be based on the size of the
reference genome, precisely, it will take at least 64 bits per unique k-mer (e.g. 1 Gbp would mean we need 8 GiB). Our
heuristic of 10 GiB per 1 Gbp of reference genome is an approximation for the overhead of the hash table, but because we
scale in powers of 2, the memory usage may be higher than this heuristic by up to a factor of 2 in the worse case.

### Adapter Designs

We have several adapter designs available in the adapter design bundle used with the `--adapter-design-bundle` option.
The following table summarizes the available adapter designs:

| Adapter Design Name | Description                                                                                |
|---------------------|--------------------------------------------------------------------------------------------|
| SBX-D               | Default duplex adapter design for paired intramolecular reads that use SBX-D chemistry.    |
| SBX-DM              | SBX-D adapter design for TAPS+ methylation libraries.                                      |
| SBX-FAST            | Default duplex adapter design for paired intramolecular reads that use SBX-FAST chemistry. |
| YSU-SL              | Simplex adapter for Linear Amplification libraries which include UMIs.                     |
| YSU-TE              | Simplex adapter for Target Enrichment Libraries which include UMIs.                        |
| YS                  | Simplex adapter for Simplex Libraries without UMIs                                         |

#### Adapter Bundle Structure

The adapter design bundle is a folder which can also be zipped into a single file for distribution.
It contains mostly fasta files and a single json file which describes the contents of the bundle.

It contains a manifest file named `manifest.json` which describes the available adapter designs in the bundle and allow
the folder structure to be flexible.

The folder structure typically looks like this:

```adapter_design_bundle/
├── adapter_name/
│   ├── adapter_feature.fasta
└── manifest.json
```

##### Adapter Bundle Manifest File Structure

The `manifest.json` file is the central configuration file that defines all available adapter designs in the bundle. It
has the following structure:

**Top-level fields:**

- `default_adapter_design_name`: The name of the adapter design to use when `--adapter-design-name` is not specified
- `adapter_designs`: An array of adapter design objects

**Adapter design object fields:**

- `name`: Unique identifier for this adapter design (used with `--adapter-design-name` like "SBX-D", "YSU-TE", etc.)
- `type`: The adapter type (e.g., "duplex", "duplex_umi", etc.)
- `adapter_5p`: Array of adapter features to search for at the 5' end of the read
- `adapter_3p`: Array of adapter features to search for at the 3' end of the read

**Adapter feature object fields:**

Each feature in `adapter_5p` or `adapter_3p` contains:

- `type`: Feature type (e.g., "runway", "sid", "anchor", "umi", "stem")
- `lut`: Lookup table configuration
  - `sequences`: Path to FASTA file containing sequences for this feature (relative to bundle root)
  - `transform`: Sequence transformation to apply ("none", "reverse_complement")
  - `max_edit_distance`: Maximum Hamming distance allowed when matching sequences
- `search`: Search strategy configuration
  - `max_wiggle_left`: Maximum bases to search left of expected position
  - `max_wiggle_right`: Maximum bases to search right of expected position

**Example snippet:**

```json
{
  "default_adapter_design_name": "SBX-D",
  "adapter_designs": [
    {
      "name": "SBX-D",
      "type": "duplex",
      "adapter_5p": [
        {
          "type": "runway",
          "lut": {
            "sequences": "duplex/runway-5p.fa",
            "transform": "none",
            "max_edit_distance": 2
          },
          "search": {
            "max_wiggle_left": 0,
            "max_wiggle_right": 2
          }
        }
      ]
    }
  ]
}
```

The manifest allows flexible organization of adapter sequences across different directories while maintaining a
consistent interface for demux to locate and use them.

### Read Name Format for FASTQ Output with RDB input

When the input files are in RDB format (a binary format), we have to regenerate the output FASTQ read names.
We support four different RDB input flavors, Gamma2, Gamma1, DataPath2, and HTP1, which will be automatically detected by the reader.
The read name outputs are generated with the following format:

```shell
{date}:{sequence}:Q{queue_name}:R{run_num}:{cycle_id}:{b64-UID}:{sidUmiBitflag}|{sidId}{optionalUMI}
```

When input files are FASTQ we preserve the original read names in the output FASTQ files, but append the `sidUmiBitflag`, `sidId`, and `optionalUMI` information to the end of the read name in the same format as above.

| Field           | Description                                                                                                                                                                                                                   |
|-----------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `date`          | Date of processing in YYYYMMDD format                                                                                                                                                                                         |
| `sequence`      | Sequence number of the read in the input file                                                                                                                                                                                 |
| `queue_name`    | Name of the queue the read was processed in. This is a `*` if not specified or identified.                                                                                                                                    |
| `run_num`       | Run number of the read.  This is a `*` if not specified or identified.                                                                                                                                                        |
| `cycle_id`      | Cycle ID of the read                                                                                                                                                                                                          |
| `b64-UID`       | Base64 encoded unique identifier of the read. Encoded using `0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz+_`, 0-padded for uniform length. Maximum size is 6 tokens.                                        |
| `sidUmiBitflag` | 4-bit bitfield printed as a base-10 integer (0-15) indicating the presence of SIDs and UMIs in the read. Bits: 5' SID found: `0b1000` (8), 3' SID found: `0b0100` (4), 5' UMI found: `0b0010` (2), 3' UMI found: `0b0001` (1) |
| `sidId`         | Assigned SID for the read                                                                                                                                                                                                     |
| `optionalUMI`   | UMI sequences if present, formatted as `:{umi5p}:{umi3p}`                                                                                                                                                                     |

### Output Directory and Filename Structure

The output directory will contain a subdirectory for each sample, named for the sample.
Each sample subdirectory will contain the FASTQ files for that sample.
The files are named as `<sample_name>.<input_filename>_<hash>[.<thread_index>].fastq[.gz|.zst]`.
The `hash` is derived from the input file path to ensure uniqueness and avoid overwriting files when using multiple writing threads per sample across multiple devices where the input files may have the same name, but reside in different directories.
When `--writing-threads-per-sample` is greater than 1, each writing thread writes to its own file suffixed with `.<thread_index>`, where `thread_index` is the zero-padded two-digit index of the thread (for example, `00`, `01`, ...). When `--writing-threads-per-sample` is set to `1`, this thread suffix is omitted from the file name.
The output directory will also contain a metrics directory with files described in the [simples metrics](#simplex-metrics) and [duplex metrics](#duplex-metrics) sections.
In addition to sample ids, if `--output-failed-reads` is set, there will be a `raw_failed` sample for unassigned reads that failed before or during SID assignment with files named `raw_failed.<input_filename>_<hash>[.<thread_index>].fastq[.gz|.zst]`.

A truncated example output directory structure may look like this with a single writing thread, when `--output-failed-reads` is set, and with two samples `HG0001` and `HG0002`:

```text
output_directory/
├── metrics/
│   ├── run_stats.tsv
│   ├── sample_stats.tsv
│   ├── sample_assignment_metrics.tsv
│   ├── trimmed_full_read_len_dist.tsv
│   ├── trimmed_partial_read_len_dist.tsv
│   ├── untrimmed_full_read_len_dist.tsv
│   └── untrimmed_partial_read_len_dist.tsv
├── HG0001/
│   ├── HG0001.PrimaryAnalysis_000001_3199979.fastq.gz
│   ├── HG0001.PrimaryAnalysis_000001_2185373.fastq.gz
│   └── HG0001.PrimaryAnalysis_000002_72005663.fastq.gz
├── HG0002/
│   ├── HG0002.PrimaryAnalysis_000001_97679291.fastq.gz
│   └── HG0002.PrimaryAnalysis_000002_58193029.fastq.gz
└── raw_failed/
    ├── raw_failed.PrimaryAnalysis_000001_97679291.fastq.gz
    └── raw_failed.PrimaryAnalysis_000001_218573.fastq.gz
```

### Genome Strand Detection

Strand detection is not required for demultiplexing, but it enables strand-aware left-align quality base shifting for
indels for duplex adapters. When strand detection is used it does not change the consensus sequence, only the location
of indels thus affecting the location of the quality scores and YC tag. It is primarily useful if duplex data is used
for tools or pipeline that are designed for Illumina data which does not have the same kind of indel profile as SBX
data. Strand detection may improve tools originally designed for Illumina data by orienting indels in a pileup the same
way at least for a read with k-mers to a specific strand of a reference genome.

Strand detection is performed via k-mer membership queries against an interleaved Bloom filter built from a reference
genome. The filter interleaves two sub-filters, one for the forward strand and one for the reverse strand to perform
both strand lookups in a single query. To minimize lookup time, we stop looking up k-mers as soon as we can classify the
read to a strand with high confidence using a binomial based test controlled by the `--strand-critical-phred` parameter.

`--strand-critical-phred` is a Phred-scaled false positive rate critical value -10*log10(critical_value). As mentioned
above, lower values will classify faster at cost of accuracy. This is because if a read is significantly classified (
reaches a count that computes into a probability that passes the probability determined by the critical value) it will
be classified significantly to a strand ceasing further k-mer look-ups early. If a read is not significantly classified
to either strand, the highest count will be used and contributes to the `strand_fw_reads` or `strand_rv_reads` metric.

Regardless of if the read is significantly classified or not, an algorithm will then take the existing pairwise
alignment of the duplex read and shift them as far right or left as possible without altering the consensus sequence.
This will result in indels being left-aligned in a pileup relative to the reference genome used to generate the
strand-index once aligned. Unclassified reads (equal count reads) will have indels shifted arbitrarily for consistency.

### Concepts and Terminology

| Term                | Description                                                                                                                                                                                                                                 |
|---------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Bloom filter        | Memory-efficient probabilistic data structure for set-membership tests.                                                                                                                                                                     |
| consensus           | A sequence generated by pairwise intramolecular consensus of two constituent reads.                                                                                                                                                         |
| duplex              | A read consisting of two constituent reads (R1, R2) that are paired intramolecularly.                                                                                                                                                       |
| full read           | A simplex read that has all expected features, including both SID and UMI sequences if they exist.                                                                                                                                          |
| partial read        | A simplex read with at least 1 SID (and 1 UMI if they exist in adapter design) identified excluding full length reads.                                                                                                                      |
| full duplex read    | A duplex read that has an identifiable insert on both R1 and R2.                                                                                                                                                                            |
| partial duplex read | A duplex read that is missing one or more expected features on either R1.                                                                                                                                                                   |
| runway              | A known fixed adapter sequence at the start of the read defined by `[runway][start adapter]`.                                                                                                                                               |
| start adapter       | Known fixed adapter sequence after the runway. On duplex reads they are defined by `[runway][start adapter][insert]`.                                                                                                                       |
| end adapter         | Known fixed adapter sequence at the end of the read. On duplex reads they are defined by `[insert][end adapter]`.                                                                                                                           |
| endadapter          | Shorthand for the duplex read adapter both start and end adapters which will share reverse complemented sequence, it flanks the insert with the structure `[endadapter][insert][hairpin]` after intramolecular consensus.                   |
| loop                | Duplex read typically 7 bp sequence sandwiched by the SID in the hairpin.                                                                                                                                                                   |
| hairpin             | Duplex read adapter sequence with structure `[SID][loop][SID]` (sometimes called midadapter). The second SID will be reverse complemented relative to the other.                                                                            |
| linker base         | A base linking adapter elements together, usually A, for example between the insert and SID in SBX-D data.                                                                                                                                  |
| stem                | A general term for an adapter sequence (usually fixed sequence) that occurs in between the endadapter sequences. If another specific name is use that usually superceeds the term stem but it is a superset of each of these adapter terms. |
| hairpin stem        | A stem section that in some adapters that may exist between the hairpin and insert `[insert][stem][hairpin]`. In SBX-D this stem is composed of only the linker base A and is more relevant on other adapter types.                         |
| partial duplex      | A duplex read with a simplex region.                                                                                                                                                                                                        |
| R1                  | First constituent read in a duplex read. A raw R1 refers to R1 sequence without adapter trimming before the loop sequence.                                                                                                                  |
| R2                  | Second constituent read in a duplex read. A raw R2 refers the R2 sequence without adapter trimming after the loop sequence.                                                                                                                 |
| SID                 | Sample ID barcode index sequence, typically 12 bp long                                                                                                                                                                                      |
| Valid SID           | SID found (within error tolerance) in specified sample sheet (via `--sample-sheet`) if no sample sheet is provided, the adapter design bundle default SIDs are used.                                                                        |
| umi                 | Unique molecular identifiers are short sequences used to uniquely tag each molecule in a sample library.                                                                                                                                    |
| YC tag              | Lossless encoding of the two constituent reads (R1, R2) of a duplex consensus read.                                                                                                                                                         |
| strand detection    | Process of determining the DNA strand (forward or reverse) relative to a reference genome                                                                                                                                                   |
| sample sheet        | A comma separated file containing SID sequences with sample names. See section [Sample Sheet Format](#sample-sheet-format) for more details.                                                                                                |
